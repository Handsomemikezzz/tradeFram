# 数据层规划

## 背景

当前量化框架优先解决研究与回测阶段的数据可用性、稳定性和可复现性问题。第一阶段先使用 AkShare 作为低成本数据源，等数据覆盖、稳定性或实时交易能力不足时，再补充或替换为 QMT、Wind、Choice、Tushare Pro 等专业数据源。

核心原则是：AkShare 只是第一版数据接入来源，不应该成为系统核心依赖。策略、回测、因子计算等上层模块只依赖内部统一数据接口和本地数据仓库。

## 总体路线

数据层采用“先全量初始化，再每日增量同步”的路线。

```text
AkShare / 其他数据源
  ↓
DataProvider 数据源适配层
  ↓
原始数据落地
  ↓
清洗与标准化
  ↓
本地数据仓库
  ↓
因子计算 / 回测 / 策略研究 / 实盘信号
```

第一阶段目标：

- 先把历史数据拉到本地。
- 每天收盘后同步最新数据。
- 每次同步支持回补最近若干交易日，避免数据源延迟或历史修正造成缺口。
- 上层研究和回测不直接调用 AkShare。

## 数据源选择

第一阶段使用 AkShare：

- 成本低，适合快速启动。
- 覆盖 A 股日 K、指数、基础信息、部分财务和宏观数据。
- 稳定性和字段一致性需要通过本地校验与缓存来兜底。

后续可扩展的数据源：

- Tushare Pro：补充更稳定的基础行情和财务数据。
- QMT / miniQMT：更适合实盘行情、账户查询、下单和撤单。
- Wind / Choice / iFinD：更适合机构级数据、指数成分、行业分类、财务和高质量历史数据。

QMT 在当前规划中暂不作为研究数据源优先级。它更偏实盘执行层，主要用于连接券商账户、获取实时行情、查询持仓资金、发送交易指令。

## 数据源适配层

需要设计统一的 `DataProvider` 边界，避免 AkShare 调用散落在业务代码中。

推荐边界：

```text
AkShareProvider
TushareProvider
QMTProvider
WindProvider
ChoiceProvider
```

上层模块只依赖统一接口，例如：

- 获取股票列表
- 获取交易日历
- 获取股票日 K
- 获取指数日 K
- 获取复权因子
- 获取财务数据
- 获取指数成分

这样未来切换数据源时，只需要新增或替换 Provider，不需要大改回测、因子和策略模块。

## 本地数据仓库

第一版建议使用：

- Parquet：存储主要行情和宽表数据。
- DuckDB：本地查询、研究分析、回测读取。
- SQLite 或轻量元数据表：存储同步状态、任务运行记录、数据版本等。

推荐目录结构：

```text
data/
  raw/
    akshare/
      daily_bars/
      index_daily_bars/
      instruments/
      trading_calendar/
  warehouse/
    daily_bars/
    index_daily_bars/
    instruments/
    trading_calendar/
    adj_factors/
  metadata/
    sync_state.db
```

`raw` 层保存接近数据源返回结果的原始数据，便于追溯和排查。`warehouse` 层保存清洗、标准化后的内部数据。

## 初始历史数据

首次初始化时，先全量拉取历史数据，建立本地数据基础。

第一阶段建议先拉：

- 股票基础信息
- 交易日历
- A 股历史日 K
- 指数历史日 K

后续再补充：

- 复权因子
- 财务数据
- 指数成分历史
- 行业和概念分类
- 停复牌状态
- ST 状态
- 涨跌停状态

日 K 数据建议同时保留：

- 未复权原始 OHLCV
- 复权因子
- 可选的前复权数据缓存

不要只保存前复权价格。未复权价格和复权因子更利于排查问题，也更方便在不同回测口径之间切换。

## 每日同步任务

每日同步任务在收盘后运行，建议先设在 16:30 或 17:00。为了处理数据源延迟或修正，可以晚上再补一次，或者每次任务都回补最近 5 到 20 个交易日。

每日流程：

```text
1. 同步或读取交易日历，判断是否交易日
2. 同步股票基础信息
3. 拉取最近 N 个交易日的 A 股日 K
4. 拉取最近 N 个交易日的指数日 K
5. 更新复权因子，若第一阶段暂未接入则预留接口
6. 执行数据校验
7. 写入本地 Parquet / 元数据表
8. 可选：触发因子重算
```

每日任务不要只拉当天数据，应该覆盖最近 N 个交易日。这样可以处理：

- AkShare 数据延迟
- 历史行情修正
- 停复牌信息变化
- 复权因子调整
- 临时缺口补齐

## 数据质量校验

第一阶段至少需要基础校验：

- 交易日是否存在数据
- 股票代码格式是否统一
- OHLC 是否为空
- 成交量和成交额是否为负
- 同一股票同一交易日是否重复
- 每日全市场股票数量是否异常
- 指数数据是否缺失
- 本次同步是否成功写入

后续可增加：

- 与备用数据源交叉校验
- 涨跌幅异常检测
- 复权前后收益一致性校验
- 停牌日成交量校验
- 数据版本快照

## 定时任务

建议拆成两个任务入口：

```text
init_history_data
sync_daily_data
```

`init_history_data` 用于首次构建本地数据仓库，通常手动执行。

`sync_daily_data` 用于每日增量同步，可以由 cron、APScheduler、系统任务或后续工作流调度器触发。

推荐调度：

```text
交易日 16:30：同步当天和最近 N 个交易日数据
交易日 20:00：可选二次回补
周末或月末：可选全量校验和重点数据重算
```

## 分阶段实施

第一阶段：

- 接入 AkShare。
- 建立 DataProvider 抽象。
- 拉取股票列表、交易日历、A 股日 K、指数日 K。
- 使用 Parquet + DuckDB 建立本地数据仓库。
- 实现每日增量同步和最近 N 日回补。

第二阶段：

- 增加复权因子。
- 增加基础财务数据。
- 增加更完整的数据质量校验。
- 建立同步状态表和任务日志。

第三阶段：

- 补充指数成分、行业分类、ST、停复牌、涨跌停等数据。
- 引入备用数据源做交叉校验。
- 评估是否接入 Tushare Pro、QMT、Wind 或 Choice。

第四阶段：

- 服务实盘信号生成。
- 将 QMT 放入实盘行情与交易执行层。
- 建立研究数据、回测数据、实盘数据之间的边界和一致性校验。

## 当前决策

- 先用 AkShare，不急于接入 QMT 或商业数据源。
- 先拉历史数据到本地，再做每日同步。
- 每日同步必须支持最近 N 个交易日回补。
- 本地优先采用 Parquet + DuckDB。
- 上层模块不直接依赖 AkShare。
- QMT 暂定位为未来实盘执行和实时行情能力，而不是第一阶段历史研究数据源。
